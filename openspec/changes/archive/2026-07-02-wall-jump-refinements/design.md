## Context

The wall-jump port (`feat/wall_jump` from `adamkl/pico-8`, ported in the prior `add-wall-jump` change) tracks `on_wall` via `player.onCollideUpdate` (sets true on any left/right overlap with a "tile") and `player.onCollideEnd("tile", ...)` (clears it only on full separation). Playtesting found two divergences from the source:

1. **Wall-riding never disengages while still touching, even after releasing the key.** Tracing the source's `chk_collision()` closely: the left/right check for a given direction is skipped only when velocity points *away* from that side (`v_x<0` skips the right-check, `v_x>0` skips the left-check). At `v_x==0` (key released, friction has decayed to zero) neither skip fires, so the check still runs — but the source's check is a **swept raycast along the current velocity vector**, evaluated in whole-pixel steps, not a persisted overlap flag. With `v_x==0`, the sweep for the wall-facing direction only advances via `v_y` (a purely vertical probe at the player's current x-offset), which can only "reach" the wall if the player is already overlapping it at that exact x — and the same stepping logic that resolves a wall collision backs the player off to the last clear whole-pixel step, so contact typically settles a fraction of a pixel short of true overlap. Net effect: with `v_x==0`, the probe can't reach the wall anymore, so `on_wall` clears — even though the player hasn't visibly moved. **Contact is regenerated by the act of moving toward the wall every frame, not persisted as state.** Kaplay's `area()`/`onCollideUpdate` does the opposite by construction: its physics step resolves overlapping bodies to genuine flush contact and reports that contact continuously regardless of current velocity or input, which is why the port never lets go.

2. **The player renders with a visible ~1px gap against walls.** Confirmed via two independent checks: (a) `player.pos.x` sits at exactly the tile-flush value while pushed into a wall (i.e. the *collision* box has zero gap), and (b) direct inspection of the sprite sheet's alpha channel shows every character frame has a fully-transparent column 0 and column 7 — the drawn art is a 6px-wide figure centered in an 8px frame. The original Lua `bbox` collision points are inset at local x=1 (left) and x=6 (right) — not 0/8 — meaning the original's collision boundary was calibrated to land where the visible art actually is. The port's `k.area()` call has no shape override, so it defaults to the full 8×8 frame, placing the collision boundary at the transparent edge instead of the visible edge — hence the gap is purely a rendering artifact of an un-ported bbox inset, not a physics bug.

## Goals / Non-Goals

**Goals:**
- `on_wall` (and therefore wall-slide and wall-jump) requires the player to actively hold the direction key into the wall for as long as they want to stay attached — releasing the key drops wall-riding immediately, even without physically separating.
- The player's collision box is inset to match the sprite's actual visible pixels, closing the rendering gap against any solid surface (not just walls).
- Keep reusing Kaplay's `area()`/`body()` collision primitives — no custom swept-raycast reimplementation.

**Non-Goals:**
- Reproducing the source's exact whole-pixel-step raycast mechanics. The fix targets the *observable behavior* (must hold the key; must render flush), not a literal port of PICO-8's swept-collision algorithm.
- Changing the inset amounts for top/bottom (ground/ceiling) collision feel — the same bbox-derived inset is applied for consistency, but no complaint was raised about vertical collision, so it's not a focus of verification.
- Re-touching `WALL_JUMP_FORCE`, `WALL_SLIDE_GRAVITY_MULT`, or any other constant.

## Decisions

**Recompute `on_wall` fresh every `onUpdate` tick as `touchingWallSide && inputTowardThatSide`, replacing the `onCollideEnd`-driven clear.** This is a pure function of this frame's collision state and this frame's input, so it can't get stuck the way an event-driven sticky flag can — there's no "did the end event fire yet" race, because there's no persisted flag to fail to clear. `onCollideUpdate` (or an equivalent per-frame `isColliding` check against "tile" objects, filtered by `isLeft()`/`isRight()`) still supplies the "touching" half; the "actively pressing" half reuses the same `goLeft`/`goRight` booleans already computed each frame for horizontal movement.

**Keep using Kaplay's overlap-based collision for the "touching" signal — don't try to replicate the source's raycast-based "just reached via motion" semantics.** The gameplay-relevant behavior the source produces (must hold the key to stay attached) is fully reproducible by gating on current input, without needing Kaplay to report "false" contact the way the source's under-shooting raycast incidentally does. Chasing the raycast mechanics literally would mean fighting `body()`/`area()`'s designed-to-be-flush collision resolution for no behavioral gain.

**Add the inset via `k.area({ shape: new k.Rect(...) })` on the player, derived from the source's bbox local points scaled by `DISPLAY_SCALE`.** The source's four bbox arrays give left=1, right=6, top=0, bottom=7 (in an 8-wide/8-tall local frame) — translating to a rect offset and size in world units after applying `DISPLAY_SCALE=4`. This is a one-time, static shape (the bbox doesn't change across animation frames in the source), so it doesn't need to vary with the current sprite frame.

**Don't trim the sprite sheet's transparent padding.** The gap is closed by moving the *collision* boundary to match where the art already is, not by re-exporting sprite frames to remove padding — that would require regenerating `scripts/extract-sprites.ts`'s embedded hex data and risks visual regressions in frames not related to this issue.

## Risks / Trade-offs

**Gating `on_wall` on current input direction, rather than reusing `flipX`/last-faced-direction, means the player must hold the *opposite* key from the one used to approach if they turn around while sliding.** → Matches the source's model (the check is against current `v_x`, not a remembered facing), and is the literal behavior being requested — not a compromise.

**The inset area() shape changes ground/ceiling collision feel slightly too (1px top inset is actually 0, so no change there; 1px bottom inset could let the player's feet visually sink 1px into the floor before the top-inset-based ground check fires)`.** → No complaint raised about vertical collision; flagged for playtest verification during implementation rather than a blocking concern.

**Recomputing `on_wall` from scratch each frame (rather than incrementally) means a single missed collision-detection frame (e.g. a physics-tick/render-tick mismatch) could cause a one-frame flicker in wall-slide state.** → Acceptable; matches the general "recompute every frame" character of the source, and Kaplay's collision events fire every physics tick, not just on transitions, so this is unlikely to be visible in practice.
